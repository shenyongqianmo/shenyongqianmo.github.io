    Redis Key

    • keys * 查看当前库所有的key
    • exists [keyname] 判断某个key是否存在
    • type [keyname] 返回key所对应的value的类型
    • del [keyname] 删除指定的key
    • unlink [keyname] 异步删除的操作
    • expire [keyname] [seconds] 为指定的key设置过期时间
    • ttl [keyname] 查看还有多少秒过期，返回剩余的秒数，如果返回-1表示永不过期，返回-2表示已过期
    • select [dbname] 切换数据库（redis默认有16个数据库，序号从0~15）（默认在0号数据库）
    • dbsize 查看库中key的个数
    • flushdb 清空当前库
    • flushall 清空所有库

    Redis List

    简介

    1. Redis List是简单的字符串列表，按照插入顺序排序。你可以添加一个元素到列表的头部或者尾部
    2. 它的底层实际是个 双向链表 ，对两端的操作性能很高，通过索引下标的操作中间的节点性能会比较差

    常用命令

    • lpush / rpush [key] [value1] [value2] [value3] … 从左边 / 右边插入一个或多个值
    • lpop / rpop [key] 从左边 / 右边吐出一个值（吐出以后，值就从该列表消失了）
    • rpoplpush [sourceList] [destination] 从sourceList中右推出一个值，然后左push到destination中
    • lrange [list] [start] [end] 按照索引下标访问指定的值（-1表示右边第一个，从0到-1表示访问所有值）
    • lindex [list] [index] 访问指定下标的值
    • llen [list] 获得列表长度
    • linset [list] before / after [value] [newvalue] 在value值的前面 / 后面插入一个newvalue
    • lrem [list] [number] [value] 从左边开始，删除number个值为[value]的元素
    • lset [list] [index] [value] 将列表指定索引的值进行替换

    Redis List 数据结构

    List的数据结构是 快速链表 quickList

    首先，列表元素较少的情况下会使用一块连续的内存，这个结构是 ziplist ，即：压缩列表

    它将所有的元素紧挨着一起存储，分配的是一块连续的内存，当数据量比较多的时候才会改成 quickList

    当数据量比较大的时候才会改成 quickList

    因为普通的链表需要的附加指针空间太大，会比较浪费空间。比如这个列表里存的只是int类型的数据，结构上还需要两个额外的指针 prev 和 next

    Redis 将链表和 ziplist 结合起来组成了 quicklist ，也就是将多个 ziplist 使用双向指针串起来使用。这样既满足了快速的插入删除性能，又不会出现太大的空间冗余

    Redis String

    简介

    1. String是Redis最基本的类型，一个key对应一个value
    2. String类型是 二进制安全 的，意味着Redis的 string 可以包含任何类型的数据，比如jpg图片或者序列化的对象
    3. String类型是Redis最基本的数据类型，一个value对应的字符串的大小最多可以是 512M

    常用命令

    • set [key] [value]
    • get [key] 查看key对应的value
    • append [key] [value] 将指定的value追加到key所对应的原字符串中，并返回新字符串的长度（如果该键不存在，则创建）
    • strlen [key] 获得字符串的长度
    • setnx [key] [value] 只有在key不存在时，才能设置value（即：不能进行值覆盖的set命令）


    • incr [key] 将key中储存的数字加1（只能对数字值进行操作，如果为空，新增值为1的字符串）
    • decr [key] 将key中存储的数字减1
    • incrby / decrby [key] [length] 将key中存储的数字按指定的步长进行增减（自定义步长，如果为空，新增值为步长的字符串）


    • mset [key1] [value1] [key2] [value2] [key3] [value3] … 同时设置多个key
    • mget [key1] [key2] … 同时获得多个值
    • msetnx [key1] [value1] [key2] [value2] … 当且仅当所有给定的key都不存在时，可以使用
    • getrange [key] [startIndex] [endIndex] 获得某个范围内的字符串切片，类似java的 substring，它是左右下标都要包含的
    • setrange [key] [startIndex] [value] 用value覆盖某个下标对应的值（索引从0开始）
    • setex [key] [expireTime] [value] 设置键值对的同时设置过期时间(expire time)
    • getset [key] [value] 类似于set的覆盖，只不过它是设置一个新value，同时返回一个老value

    数据结构

    String的数据结构为简单动态字符串(Simple Dynamic String,缩写SDS)。是可以修改的字符串，内部结构实现上类似于Java的ArrayList，采用预分配冗余空间的方式来减少内存的频繁分配。

    如图中所示，内部为当前字符串实际分配的空间capacity一般要高于实际字符串长度len。当字符串长度小于1M时，扩容都是加倍现有的空间，如果超过1M，扩容时一次只会多扩1M的空间。需要注意的是字符串最大长度为512M。

      Redis 客户端可以订阅任意数量的频道。

    Redis Bitmaps

    介绍

    现代计算机用二进制（位） 作为信息的基础单位， 1个字节等于8位， 例如“abc”字符串是由3个字节组成， 但实际在计算机存储时将其用二进制表示， “abc”分别对应的ASCII码分别是97、 98、 99， 对应的二进制分别是01100001、 01100010和01100011，如下图

    合理地使用操作位能够有效地提高内存使用率和开发效率。

    （1）Bitmaps本身不是一种数据类型， 实际上它就是字符串（key-value） ， 但是它可以对字符串的位进行操作。

    （2）Bitmaps单独提供了一套命令， 所以在Redis中使用Bitmaps和使用字符串的方法不太相同。 可以把Bitmaps想象成一个以位为单位的数组， 数组的每个单元只能存储0和1， 数组的下标在Bitmaps中叫做偏移量。

    命令

    名称 描述
    setbit [key] [offset] [value] 设置Bitmaps中某个位置的值
    getbit [key] [offset] 获取Bitmaps中某个位置的值
    bitcount [key] [start] [end] 统计字符串被设置为1的bit数，一般情况下，给定的整个字符串都会被进行计数，通过指定额外的 start 或 end 参数，可以让计数只在特定的范围内进行
    bitop [and/or/not/xor] [newkey] [key…] bitop是一个复合操作， 它可以做多个Bitmaps的and（交集） 、 or（并集） 、 not（非） 、 xor（异或） 操作并将结果保存在destkey中

    Bitmaps和set对比

    假设网站有1亿用户， 每天独立访问的用户有5千万， 如果每天用集合类型和Bitmaps分别存储活跃用户可以得到表

    set和Bitmaps存储一天活跃用户对比<

    数据类型 每个用户id占用空间 需要存储的用户量 全部内存量
    集合类型 64位 50000000 64位*50000000 = 400MB
    Bitmaps 1位 100000000 1位*100000000 = 12.5MB

    很明显， 这种情况下使用Bitmaps能节省很多的内存空间， 尤其是随着时间推移节省的内存还是非常可观的

    set和Bitmaps存储独立用户空间对比

    数据类型 一天 一个月 一年
    集合类型 400MB 12GB 144GB
    Bitmaps 12.5MB 375MB 4.5GB

    但Bitmaps并不是万金油， 假如该网站每天的独立访问用户很少， 例如只有10万（大量的僵尸用户） ， 那么两者的对比如下表所示， 很显然， 这时候使用Bitmaps就不太合适了， 因为基本上大部分位都是0

    set和Bitmaps存储一天活跃用户对比（独立用户比较少）

    数据类型 需要存储的用户量 全部内存量
    集合类型 100000 64位*100000 = 800KB
    Bitmaps 100000000 1位*100000000 = 12.5MB
